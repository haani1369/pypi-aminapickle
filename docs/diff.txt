diff engine


purpose

    compare the sdist file tree against the source repo file tree and
    report the differences that matter for provenance: files the
    sdist ships that the repo does not have, and files whose content
    the sdist changed relative to the repo. these are the code-
    injection surface. the tool reports these; it does not judge
    intent.

    module: src/pypi_aminapickle/diff.py

    both inputs are mappings from posix relative path to lowercase
    hex sha256, produced by sdist_files and repo_files, which share
    the same path and digest conventions so paths line up directly.


what is compared, what is ignored

    for each path present in the sdist tree, after excluding the
    build-generated paths below:

        * present in the repo with an equal digest -> match, no
          finding.
        * present in the repo with a different digest -> an "altered"
          finding.
        * absent from the repo -> an "extra" finding.

    paths present only in the repo are not findings. sdists routinely
    omit repo files (tests, ci config, docs); an omission is not a
    provenance failure for what is being installed.

    build-generated paths are excluded from the sdist side before
    comparison, because they are produced by packaging and never
    exist in the repo. a path is build-generated if:

        * it is exactly "PKG-INFO", or
        * any of its "/"-separated components ends with ".egg-info".


setup.cfg normalization

    the digests the trees carry are content-normalized for one file,
    setup.cfg, because setuptools rewrites it when building an sdist:
    it re-serializes the file through configparser (dropping comments
    and changing continuation indentation) and appends a generated
    "[egg_info]" section. a byte comparison of setup.cfg therefore
    reports a difference on packages that did not change anything.

    to avoid that false positive without hiding real changes, the
    digest of a file named setup.cfg is computed over a canonical
    form: parse it with configparser, drop the "[egg_info]" section,
    and hash the sorted (section, key, value) mapping. this is
    applied identically to the sdist and the repo side (in
    sdist_files and repo_files), so a setup.cfg that only differs by
    setuptools' reformatting matches, while any real change to a
    value (a dependency, an option) still changes the digest and is
    reported. a setup.cfg that does not parse falls back to a plain
    byte digest.

    a setup.py-based project usually has no setup.cfg in its repo,
    but its sdist gets one containing only the generated [egg_info]
    section. such a setup.cfg canonicalizes to an empty config; a
    sdist-only setup.cfg with that empty digest is treated as build-
    generated and not reported as an extra file. a sdist-only
    setup.cfg carrying real configuration is still reported.


public interface

    Finding (frozen dataclass)
        kind: str    "extra" or "altered"
        path: str    the posix relative path

    diff_trees(sdist: dict[str, str], repo: dict[str, str])
            -> list[Finding]
        return the findings for the two trees, sorted by path then
        kind, so the output is deterministic. an empty list means the
        sdist matches the repo (every shipped file is present and
        unchanged). the caller turns a non-empty list into a mismatch
        verdict.


testing

    offline, over in-memory dicts:

        * identical trees produce no findings.
        * a path only in the sdist produces one "extra" finding.
        * a shared path with differing digests produces one "altered"
          finding.
        * a path only in the repo produces no finding.
        * a sdist-only "PKG-INFO" produces no finding.
        * sdist-only paths under a "*.egg-info" component produce no
          finding.
        * a mix yields all expected findings in sorted (path, kind)
          order.
        * an altered file that would also look extra is reported once
          as altered, not extra.
